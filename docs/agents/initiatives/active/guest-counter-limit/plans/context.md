**`post` handler (full):**

```ts
export const post = async (req: Request<{}, {}, CreateCounterRequest>, res: Response<CounterResponse>) => {
    try {
        const userId = req.user?.id;
        const { id, title, count, color, type, inviteCode } = req.body;

        if (!userId) {
            return res.status(BAD_REQUEST).json({ success: false, message: 'Invalid userId' });
        }

        const counter = await counterRepository.post({ id, userId, title, count, color, type, inviteCode });

        if (!counter) {
            return res.status(NOT_FOUND).json({ success: false, message: 'Counter not found' });
        }

        res.status(CREATED).json({
            success: true,
            message: 'Counter created successfully',
            data: { counter },
        });
    } catch (error: any) {
        console.error('Counter Controller Error: ', error);
        res.status(SERVER_ERROR).json({ success: false, message: 'Server error: ' + error.message });
    }
};
```

**Where to insert the guard:**  
After the `!userId` check (line 20–22), before `counterRepository.post(...)` is called. The guard requires:

1. Fetching the user's tier from DB (via `userRepository.getUserById(userId)`)
2. Counting owned counters (new repository function on `counterRepository`)
3. If `tier === 'BASIC'` and count `>= MAX_FREE_COUNTERS`, return `res.status(FORBIDDEN).json(...)`

**Critical note — `req.user` does NOT contain `tier`:**

There is no evidence that `tier` is in the JWT payload. **The server must do a DB query to get the user's current tier** — this is consistent with the architectural decision already made in the plan.

## 12. Client — Counter Service (Supporting Context)

### File: `app/client/src/services/counter.service.ts`

`CounterService.create(counter)` enqueues a sync command, does **not** call the API directly:

```ts
async create(counter: ClientCounter) {
    await SyncQueueService.addCommand({ type: 'CREATE', entity: 'counter', ... });
    SyncManager.processQueue();
}
```

- This means if the server returns 403, it is handled inside `SyncManager.processQueue()`, not directly in `counterStore.createCounter`.
- **Complication:** The client-side optimistic update (push to `counters.value`) happens before the server response. If the server rejects with 403, the client store will have an extra counter that the server doesn't. This is an existing architectural pattern (offline-first) — the limit guard in HomeView is the primary protection to prevent users from ever reaching this state.

## Key Cross-Cutting Gotchas

1. **`req.user.tier` reliability:** The JWT payload may or may not include `tier`. Always DB-query for user tier in the server guard — do not trust `req.user.tier`.

2. **`countByOwner` must be mocked in existing test file:** `vi.mock('../../../db/repositories/counter.repository', ...)` lists every function explicitly. The new `countByOwner` function must be added to that mock or existing tests will break.

3. **`userRepository` import in counter controller:** The controller currently only imports `counterRepository`. Adding `userRepository` is a new import coupling — acceptable per plan, but note it adds a second DB call on every `POST /counters`.

4. **Client `counters.length` includes shared counters:** Use `counterStore.counters.filter(c => c.userId === authStore.user?.id).length` for the client-side check to match the server's `countByOwner` behavior.

5. **ESM `.ts` extension in barrel files:** When adding `export * from './limits.ts'` to the constants barrel, the `.ts` extension is required (matches existing pattern in `index.ts`).

6. **`IonModal` not yet in HomeView:** Must add `IonModal` (and likely `IonModalContent`) to the `@ionic/vue` import list.

7. **`/upgrade` route has no guard:** Per plan, it's a standalone route. If it should only be accessible to non-premium users, a navigation guard or in-component redirect would be needed. The plan is silent on this — Low-Level Planner should decide.

8. **`MAX_FREE_COUNTERS` import on client:** Client imports `@packages/utils` the same way as server (both are in the same monorepo and the package is ESM with direct TS source exports). Verify client `tsconfig/vite` aliases support `@packages/utils`.
